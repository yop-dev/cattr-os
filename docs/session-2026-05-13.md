### BUG-011 — Desktop timer resets to 00:00:00 when syncing from web

**Symptom:** When a timer was started on the web bar, the desktop app's tracker card showed the wrong elapsed time — `00:00:31` while the web showed `00:01:54` for the same session. Stopping and starting multiple times made the discrepancy worse.

**Root cause:** `Tracker.vue._startSessionTimer()` always anchored the session timer to `Date.now()` — the moment the desktop received the sync signal. It had no knowledge of the server session's `start_at`. The web bar correctly calculates elapsed time as `Date.now() - session.start_at`. The desktop calculated `Date.now() - Date.now()` (effectively 0), so it always started from `00:00:00` when syncing in.

This affected:
- Web → Desktop sync (most visible case)
- Task switches detected externally
- Any case where the desktop joined a session already in progress

**Fix:** Threaded the server's `start_at` from `web-sync.js` through the IPC event chain into the store, then used it in `Tracker.vue` as the timer anchor.

#### Data flow after fix

```
web-sync.js          → sets _externalStartAt = srv.start_at before TaskTracker.start()
task-tracking.js     → includes startAt: webSync.getExternalStartAt() in tracking/event-started IPC event
User.vue             → commits setTrackingStartAt(startAt) to store on event-started; clears on event-stopped
Tracker.vue          → _startSessionTimer() reads store.trackingStartAt; uses it as anchor if present, else Date.now()
```

For desktop-native starts (user clicks ▶ in the app), `getExternalStartAt()` returns `null` → `Tracker.vue` falls back to `Date.now()` → behavior unchanged.

|---|---|---|

| `app/src/routes/task-tracking.js` | desktop-application | Required `web-sync`; added `startAt: webSync.getExternalStartAt()` to `tracking/event-started` IPC event (both `started` and `switched` handlers) |

#### Key technical note — timing of `_externalStartAt`

`web-sync.js` sets `_externalStartAt = srv.start_at` immediately before `await TaskTracker.start()`. The `'started'` event fires synchronously inside `TaskTracker.start()`. `task-tracking.js`'s event listener runs at that point and reads `getExternalStartAt()` — the value is guaranteed to be set. After the await, `_externalStartAt` is cleared to `null`.

This means `getExternalStartAt()` returns a non-null value only during the synchronous window between set and clear — never leaks into desktop-native start paths.

### BUG-012 — Screenshots page shows wrong date (or no screenshots) compared to Dashboard

**Symptom:** Screenshots taken at 11 PM+ local time (PDT) appeared on the Dashboard under the correct date (e.g. May 13) but were invisible on the Screenshots page for that same date. After an initial attempted fix, they appeared but under the previous date (May 12) instead of May 13.

**Root cause — two compounding issues:**

**1. `getSelectedDate()` returned the wrong date for UTC-midnight Date objects**

AT-UI's datepicker stores the selected date as a JavaScript `Date` object. When the user picks "May 13", the object may represent `new Date('2026-05-13')` = May 13 00:00 **UTC** (not local midnight). Formatting that with `Intl.DateTimeFormat` in the `America/Los_Angeles` timezone converts it to May 12 17:00 PDT → returns `"2026-05-12"`, one day behind.

Fix: use `d.toISOString().slice(0, 10)` for Date objects, which always reads the UTC calendar date and is immune to timezone shifts.

**2. `fetchIntervals` used local-timezone day boundaries instead of UTC**

The original `dayBoundsUtc(dateStr, tz)` function converted the selected date string ("2026-05-13") into UTC bounds using the company timezone (PDT = UTC−7):
- Start: May 13 00:00 PDT = May 13 07:00 UTC
- End: May 13 23:59 PDT = May 14 06:59 UTC

Screenshots captured at 11:27 PM PDT May 12 have a server-side UTC timestamp of **May 13 06:27 UTC**. This falls **before** the 07:00 UTC start bound → excluded from the query → "No screenshots found."

The Dashboard's native Cattr component uses **UTC midnight-to-midnight** boundaries, so it correctly includes 06:27 UTC May 13 under "May 13."

**Fix:** Replaced `dayBoundsUtc` with plain UTC day bounds: `[dateStr + ' 00:00:00', dateStr + ' 23:59:59']`. The `dayBoundsUtc` function was removed entirely.

#### Why the two-step fix was needed

The first fix (Date object timezone issue) alone caused screenshots to shift to the "May 12" view: `getSelectedDate()` now correctly returned "2026-05-13", but the PDT-based query window (starting at 07:00 UTC) still excluded the 06:27 UTC screenshots. Only when the user navigated to "May 12" did the query window (May 12 07:00 UTC – May 13 06:59 UTC) accidentally include the screenshots. The second fix (UTC boundaries) resolved this.

| File | Repo | Change |
|---|---|---|
| `app/public/screenshots-grouped.js` | cattr-server | `getSelectedDate()`: use `d.toISOString().slice(0,10)` for Date objects; removed `dayBoundsUtc`; `fetchIntervals` now uses UTC midnight bounds directly |

### BUG-013 — Web-started session records only partial duration when stopped from desktop

**Symptom:** Sessions started on the web and stopped on the desktop recorded a fraction of the actual duration. A 7-minute session recorded only 42 seconds ("eyes" test). Follow-up tests showed similar under-counting plus duplicate/unsynced SQLite intervals.

**Root cause — three compounding issues:**

**1. No gap interval for pre-detection period**

`pollOnce()` calls `TaskTracker.start()` the moment it detects a web session. Time already elapsed before desktop detection was never recorded. The server's `TrackingSessionController.stop()` only calls `Cache::forget()` — it creates no interval. The desktop is the sole interval logger.

**2. Concurrent gap pushes from 1-second `setInterval`**

`pollOnce()` is async but `setInterval` doesn't await it. While poll #1 was awaiting `pushGapInterval()` (desktop not yet started, so `desktopTracking=false`), polls #2, #3, … also saw `desktopTracking=false` and each pushed their own progressively-larger gap interval. "Ears" test produced 4 overlapping gap intervals.

**3. STOP_DEBOUNCE false-fires from `api.post` returning `{success:false}`**

`api.post` in `@cattr/node` never throws — returns `{success:false}` on all network/API errors. The old code treated any `{success:false}` as "no active session" and incremented `_externalStopCount`. After 2 consecutive network blips, `STOP_DEBOUNCE` fired: desktop stopped, `_lastPushedGapStartAt` was cleared, web session reappeared on next good poll, and a new larger gap was pushed. This cycle repeated every ~60s creating cascading duplicates in SQLite.

**Fix — three changes to `app/src/base/web-sync.js`:**

1. **`pushGapInterval()`** — new function; called when gap > 30s: pushes a catch-up interval from `srv.start_at` to `Date.now()` via `IntervalsController.pushTimeInterval()` before `TaskTracker.start()`.

2. **Mutex + session guard** — `_externalStartInProgress` flag prevents concurrent polls from running external-start logic simultaneously. `_lastPushedGapStartAt` (set to `srv.start_at` before the `await`) prevents the same session's gap from being pushed more than once. Intentionally NOT cleared on external stop — only cleared when the user stops natively (signals a new session is coming).

3. **Skip poll on `api.post` failure** — `if (!res || !res.success) return` at the top of `pollOnce()`. STOP_DEBOUNCE now only fires on definitively successful API calls that return null data.

**Verification ("mouth" test, taskId=36):**

| remoteId | startAt→endAt (UTC) | Duration | Type |
|---|---|---|---|
| 126 | 16:50:17→16:53:19 | 3:01 | Gap catch-up |
| 127 | 16:53:20→16:56:20 | 3:00 | Regular periodic |
| 128 | 16:56:21→16:57:15 | 0:53 | Final on stop |

All `synced=1`, no duplicates, no unsynced rows. Total ~6:54 for a ~7-minute session. ✅

### BUG-014 — Screenshots page shows blank card for stop interval (pending)

**Symptom:** After a web-started session is stopped from the desktop, the Screenshots page shows a third card for the stop interval (interval 128, ~53 seconds). The thumbnail area is blank white. The user expected 2 cards for the ~7-minute "mouth" test session.

**Root cause (upstream):** Cattr's `has_screenshot` model accessor returns `true` for ALL intervals when `screenshot_id=NULL` — which applies to every interval in our setup, because the `sus_files` table is empty and screenshots are stored on disk by `sha256(interval.id)` naming without any DB record. The `!null=true` short-circuit means `Storage::exists()` is never called.

**Why client-side fixes didn't work:** After four attempts, the card persists. The most likely explanation is that the thumbnail file for interval 128 **actually exists on disk** — a screenshot was captured during the 53-second stop window. The thumbnail endpoint returns HTTP 200 with a valid JPEG, so the onError/sweep/blob-size checks all pass. The "blank white" appearance may be the real screenshot content (e.g. a white screen or minimized window at the moment of capture).

#### Four fix attempts applied to `screenshots-grouped.js`

1. Filter `_allIntervals` by `has_screenshot` before rendering — did nothing; all intervals have `has_screenshot=true`
2. `onError` callback in `apiFetchImage` to remove card on non-2xx response — never triggered if endpoint returns 200
3. `setTimeout(3000)` sweep checking `!img.dataset.blobUrl` — never triggered if blob URL was set on success
4. `blob.size < 100` guard (treat empty blob as failure) + sweep also checks `img.style.display === 'none'` — still not resolved; if the JPEG is a real file its size is > 100 bytes

## Key Technical Notes

### `_externalStartAt` is a synchronous window variable

`web-sync.js` relies on the fact that `TaskTracker`'s `'started'` event fires synchronously inside `await TaskTracker.start()`. The variable is set just before the call and cleared just after. The event listener in `task-tracking.js` reads it during that synchronous window. This is safe because Node.js is single-threaded — no concurrent access is possible.

### Desktop timer anchor: store beats local clock

`Tracker.vue._startSessionTimer()` now checks `this.$store.getters.trackingStartAt` first. If the store has a server-provided `start_at`, that becomes `sessionStartMs`. The `sessionSeconds` counter is seeded immediately as `Math.floor((Date.now() - sessionStartMs) / 1000)` rather than 0, so the display is correct on the very first render tick — no 1-second jump.

### Desktop rebuild required

All BUG-011 changes are in the desktop app (`C:\desktop-application\`), not the Docker image. The Docker image serves the web app only. To apply these changes, the desktop app must be rebuilt with `npm run build` / `electron-builder`.

### Screenshots page date grouping: UTC, not local timezone

Cattr stores all `start_at` timestamps in UTC. The Dashboard's native component groups intervals by UTC date. Our custom Screenshots page now does the same — using `[dateStr + ' 00:00:00', dateStr + ' 23:59:59']` as the UTC query window. This means a screenshot captured at 11 PM PDT (= 06:xx UTC next day) is grouped under the UTC date (next day), which matches the Dashboard. If the company ever needs local-timezone grouping, `dayBoundsUtc` would need to be reinstated — but for now UTC alignment with Dashboard is the correct behavior.

### `new Date('YYYY-MM-DD')` is UTC midnight, not local midnight

JavaScript parses ISO date-only strings (`'2026-05-13'`) as UTC midnight per spec. `new Date('2026-05-13')` = May 13 00:00 UTC. When formatted with a negative-UTC-offset timezone (e.g. `America/Los_Angeles`, UTC−7), this becomes May 12 17:00 local — one calendar day behind. Always use `d.toISOString().slice(0, 10)` to read the UTC calendar date from such a Date object, not `Intl.DateTimeFormat` with a local timezone.
