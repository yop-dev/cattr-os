## What Was Done

### C-022 — Hide In-Progress Intervals

**Status:** ✅ Done and confirmed working.

**Problem:** The desktop agent pushes periodic 3-minute intervals to the server during an active tracking session. These appear immediately in the DB and show up in Reports, Screenshots, and the Dashboard sidebar while the timer is still running — confusing, since the session isn't finished yet.

**Solution:** Client-side filter using `POST /api/tracking/current`. After fetching intervals (or reading from the Vuex store), call the endpoint to get the active session. Strip any intervals belonging to it before rendering.

**Filter logic:** An interval is excluded when ALL of: (1) belongs to the current user, (2) task matches the active session's `task_id`, (3) `start_at >= session.start_at`. If no session is active, all intervals show normally.

### Dashboard Sidebar — Per-Interval Rows (matching Reports format)

**Status:** ✅ Done.

**Problem:** After C-022 landed, the Dashboard sidebar showed an aggregated time range (e.g., "12:56 AM – 1:33 AM") across all intervals for the day. This was misleading — it implied continuous work when there were gaps. User wanted it to match the Reports page format: each session shown as its own row with bold duration and gray time range below.

**Solution (iterated):**

1. **Removed the aggregated range** — replaced `taskRanges` (start/end tracking) with per-interval rows.
2. **Added `fmtTime()` / `fmtDur()` helpers** for formatting timestamps and durations.
3. **CSS**: Added `flex-wrap: wrap` to `li.task`; added `.dn-iv-list`, `.dn-iv-row`, `.dn-iv-dur`, `.dn-iv-range` styles; changed `.dn-iv-row` to `flex-direction: column` so duration stacks above time range (matching Reports layout).
4. **Merge contiguous intervals**: Initially tried merging from the Vuex store data with a 30s threshold — failed because Vuex timestamps have slightly lower precision than API responses (32s gap wasn't merging). Bumping to 60s over-merged (collapsed separate sessions).
5. **Switched to API data source**: Replaced Vuex store reading with a direct call to `POST /api/time-intervals/list` using the exact same QueryHelper format as `timecard-export.js` (with `with`, `where`, `orderBy`, `perPage`). Applied the exact same 30s merge. Results now always match Reports.
6. **Cache with session-change invalidation**: `_dnSidebarCache` (15s TTL); `_dnLastSessKey` tracks session start/stop — clears cache on change so stopped session's intervals appear immediately.
7. **Immediate injection after fetch**: Without the timebar's D3 DOM mutations, the MutationObserver no longer fires reliably after an async fetch. Fixed by calling `injectSidebarTimes()` directly from the fetch callback instead of waiting for the observer.

### BUG-025 — Desktop Duplicate Intervals Fix

**Status:** ✅ Fixed and DB-verified.

**Problem:** A ~1–2 minute desktop session intermittently produced 3 DB intervals instead of 1: two with identical `start_at` + one short tail. Root cause: `stop()` sets `this.active = false` only **after** `await captureCurrentInterval()` completes (50–500ms async window). Any concurrent caller reaching `captureCurrentInterval()` in that window passed the `!this.active` guard and pushed a duplicate. A display patch (`mergeContiguousIntervals`) was hiding this in Reports but dirty data continued accumulating.

**Fix:** Three concurrency guards added to `app/src/base/task-tracker.js` in `desktop-application`:

1. **`_stopInProgress` mutex on `stop()`** — blocks concurrent re-entry; returns `false` immediately if already stopping; `finally` guarantees flag reset even if `captureCurrentInterval()` throws
2. **Return-value check in `start()`'s task-switch path** — throws `UIError(409)` if the internal `stop(true, false)` call is blocked, preventing silent state corruption where the original `stop()`'s cleanup would overwrite the new session's state
3. **`_captureInProgress` mutex on `captureCurrentInterval()`** — blocks concurrent captures from any path including the floating (unawaited) `interval-capture` event handler; `finally` guarantees flag reset

**Verified:** 3 test sessions (32s, 91s, task-switch) — DB shows exactly 2 rows, no duplicates, no tail.

### EC-001 — "Too Many Attempts" errors / occasional logout

**Status:** 🔴 Open (documented in `edge-cases-tracker.md`)

The rate limit on tracking routes (`throttle:600,1`) is per-IP. With multiple users behind the same office NAT, or with multiple tabs/devices polling simultaneously, the shared bucket can be exhausted. Fix: switch to per-user throttling in `app/routes/api.php` using `RateLimiter::for('tracking', ...)` keyed by `$request->user()->id`.
