### C-012 — Time Interval Form Fixes (iterative, continued from 2026-05-06)

Several iterations were needed to get the task search UX fully working. All changes are in `app/public/time-interval-helpers.js`.

#### Issue 1: "Sorry, no matching options" on focus

`onSearch` was changed to call `fetchTasks(query, loading)` unconditionally (including empty query). The backend returns nothing for empty string searches, which wiped `lazySelect.options` every time the field received focus. Fixed by reverting `onSearch` to `query.length >= 1` and adding a separate `loadInitialTasks()` function for the focusin handler.

#### Issue 2: White space in dropdown on focus

`loadInitialTasks()` was calling `POST /api/tasks/list` and setting `lazySelect.options = data.data` directly. The dropdown showed items but with blank text.

**Root cause (found by grepping compiled `/app/public/dist/app.js`):** `LazySelect.fetchTasks` maps raw API task objects to add a computed `label` field before setting options:
```javascript
{ ...task, label: `${task.task_name} (${task.project.name})` }
```
v-select renders using `label`, not `task_name`. Setting raw API objects without this field produced blank entries.

**Fix:** Added `labelledTask(task)` helper that mirrors what `fetchTasks` does. All code that sets `lazySelect.options` directly now runs through `labelledTask()`. The API request also includes `with: ['project']` so project name is available.

#### Issue 3: Placeholder "Search tasks…" visible behind selected task

`patchPlaceholder()` was setting `.vs__search` placeholder to `'Search tasks…'` every tick unconditionally. v-select keeps the search input in the DOM even after selection — our tick kept overwriting the placeholder to be visible.

**Fix:** Check `!!lazySelect.$el.querySelector('.vs__selected')` instead of `lazySelect.value` (internal prop name is unreliable). Set placeholder to `''` when a task is selected, `'Search tasks…'` when empty.

#### Issue 4: Newly created task didn't auto-populate when another task was already selected

`vSelectComp.select(option)` was replaced with `lazySelect.inputHandler(newTask.id)` in an earlier fix — `inputHandler` updates the parent form's reactive data but does NOT update v-select's visual state. The field appeared to ignore the new task.

**Fix:** Reverted to `vSelectComp.select(labelledTask(newTask))`. This triggers the full event chain: v-select emits `input` → LazySelect calls `inputHandler` → parent form updates. Both visual state and form data update correctly. The earlier bug with this approach was caused by passing a custom option object without the `label` field — now fixed by `labelledTask()`.

#### Issue 5: Can't select existing tasks / can't remove after creation

After task creation with `vSelectComp.select()`, `_dn_initial_loaded` is reset to `false`. On next focus, `loadInitialTasks()` fires and reloads the full task list (which now includes the newly created task). This restores normal dropdown behaviour.

### C-013 — Timecard Export

New feature: the Time Use Report page now shows a Clockify-style per-interval table instead of Cattr's native aggregated accordion. Implemented as a new standalone script `app/public/timecard-export.js` (~340 lines).

#### What the page shows

- **Heading:** "Detailed report" with date range and total duration
- **Table columns:** Date | Description (task name + project) | Duration (HH:MM:SS + start→end time slot) | User
- **Export PDF button** — generates a PDF client-side using jsPDF + jspdf-autotable (loaded on-demand from jsDelivr CDN) and triggers a direct file download (Windows Save As dialog). Falls back to `window.print()` if CDN load fails. Filename: `Cattr_Time_Report_Detailed_{start}-{end}.pdf`.
- Company timezone applied to all timestamps via `Intl.DateTimeFormat.formatToParts`.

#### Architecture

The native Cattr Time Use Report page (`/report/time-use`) has no per-interval export. Its backend only returns aggregated data (`POST /api/report/time`). Our script bypasses this and calls `POST /api/time-intervals/list` directly with `with: ['task', 'task.project', 'user']` to get individual interval records.

The script uses a `MutationObserver` (same pattern as other injected scripts). On each tick it checks if we're on `/report/time-use`, injects the container, and only auto-refetches when the **date range** changes. User filter changes require an explicit Apply button click.

#### Bug fixed: page freeze (MutationObserver re-entrancy)

**Symptom:** Clicking Reports caused the page to freeze.

**Root cause:** `renderTimecard()` set `currentStart/End` after an `await`, but wrote to `container.innerHTML = '<p>Loading…</p>'` first. That DOM write triggered `MutationObserver → tick() → renderTimecard()` again. The second call saw stale `currentStart/End` (not yet updated), so it passed the "dates changed?" check and triggered another fetch — infinite loop.

**Fix:** Set `_fetching = true`, `currentStart`, and `currentEnd` ALL BEFORE the first DOM write:

```javascript
async function renderTimecard() {
    if (_fetching) return;
    var dates = getSessionDates();
    if (!dates.start || !dates.end) return;
    var userIds = getSelectedUserIds();
    // ALL state BEFORE any DOM write
    _fetching    = true;
    currentStart = dates.start;
    currentEnd   = dates.end;
    var container = document.getElementById(CONTAINER_ID);
    if (!container) { _fetching = false; return; }
    container.innerHTML = '<p class="dn-tc-loading">Loading…</p>'; // safe now
    // ...
}
```

#### Bug fixed: user filter showing wrong results

**Symptom:** Selecting Admin showed no time. Selecting nothing showed Admin's time. Selecting Alice + Bob showed only Alice.

**Root cause:** Cattr's `QueryHelper` parses a `where` clause value as `[$operator, $value]`. Passing `user_id: [1, 2]` was interpreted as `$operator=1, $value=2` → `WHERE user_id = 2` (only the second user). For a single-element array `[1]`: `$operator=1, $value=null` → malformed query → no results.

**Fix:** Wrap the ID array in the correct two-element format:
```javascript
// Before (broken):
where['user_id'] = userIds;
// After (correct):
where['user_id'] = ['=', userIds]; // QueryHelper: $operator='=', $value=[...] → whereIn
```
The `=` operator with an array value hits the `whereIn` branch in QueryHelper.

#### UX: Apply button for user filter

`injectApplyButton()` adds `<button id="dn-apply-filter-btn">Apply</button>` into `.time-use-report .controls-row`. On click it sets `currentStart = null` (forces a date mismatch) then calls `renderTimecard()`. This separates user filter changes from date changes — date changes auto-refetch immediately; user filter changes require explicit Apply. Prevents accidental mid-selection refetches (e.g. partial multi-user selection fetching after each click).

#### "Timecard Export" heading suppressed

`patchReportI18n()` renames the i18n key that the page `<h1 class="page-title">` renders. That heading would have shown "Timecard Export" on the page — redundant with our custom header. Suppressed via CSS: `.time-use-report .page-title { display: none !important; }`.

### C-013 continued — UX polish and bug fixes (second half of session)

#### Non-admin users auto-filtered to own time

Added `isAdmin()` (reads `parseInt(user.role_id, 10) === 0` from Vue store) and `getCurrentUserId()` helpers. In `renderTimecard()`:

```javascript
var userIds = isAdmin()
    ? getSelectedUserIds()
    : (function () { var id = getCurrentUserId(); return id ? [id] : []; }());
```

Non-admins always fetch only their own intervals regardless of the Vue user filter state. The Apply button is admin-only.

#### Apply button moved into dropdown portal

Multiple attempts to place the Apply button adjacent to the user select in the controls row all failed — the AT-UI dropdown is portaled to `<body>` and the controls row uses an opaque flex layout that doesn't accept external DOM insertion predictably.

Final approach: inject the Apply button **inside the dropdown itself** at the bottom. The function `cleanupUserDropdown()` (called from `tick()`) finds every open `.at-select__dropdown`, hides the Active/Inactive tabs and role-filter select, and injects the Apply button:

```javascript
if (isAdmin() && !d.querySelector('#' + APPLY_BTN_ID)) {
    var btn = document.createElement('button');
    btn.style.cssText = 'width: calc(100% - 16px); margin: 8px; display: block;';
    d.appendChild(btn);
}
```

#### Active/Inactive tabs and role filter hidden

The dropdown's Active/Inactive tabs and the All/Employee/Client role filter are UI we don't need. CSS selectors couldn't reach the portaled dropdown, so JS is used: `cleanupUserDropdown()` finds elements by text content (`'Active'`, `'Inactive'`) and hides them, and also hides any `.at-select` inside the dropdown (the role filter).

#### Bug fixed: Apply reads stale Vue selection (needs two clicks)

**Symptom:** First Apply click showed old data; second click worked.

**Root cause:** AT-UI multi-select only commits the selection to the parent Vue component's `userIDs` when the **dropdown closes**. Clicking Apply (inside the dropdown) read `userIDs` before the dropdown closed — getting the previous selection.

**Fix:** Close the dropdown first, then read the selection after 100ms:
```javascript
btn.addEventListener('click', function (e) {
    e.stopPropagation();
    document.body.click(); // close dropdown → Vue commits userIDs
    setTimeout(function () {
        currentStart = null;
        renderTimecard();
    }, 100);
});
```

#### Bug fixed: parent route component's `userIDs = []` returned instead of leaf component's

**Symptom:** Admin selected only Alice but all users' time showed.

**Root cause:** `getSelectedUserIds()` iterated `vm.$route.matched` shallow-to-deep. A parent route component also had a `userIDs` property (empty `[]`) and was found first. With `userIds = []`, `fetchIntervals` added no user filter → all users returned.

**Fix:** Search deepest-to-shallowest so the `TimeuseReport` leaf component is found first:
```javascript
for (var i = matched.length - 1; i >= 0; i--) { ... }
```

#### Bug fixed: regular user data disappears on page reload

**Symptom:** First navigation to Reports showed user's data. Page reload showed nothing. Navigate away and back fixed it.

**Root cause:** On reload, Vue re-renders the `.time-use-report` component a second time after auth state loads (Vuex update). This destroys our injected container. `injectContainer()` re-creates it empty, but `tick()` doesn't re-fetch because `dates.start === currentStart` (dates didn't change).

**Fix:** In `injectContainer()`, reset `currentStart/currentEnd` to null whenever the container is (re)created:
```javascript
page.appendChild(div);
currentStart = null; // force refetch on next tick
currentEnd   = null;
```

## Key Technical Notes

### v-select `label` field (confirmed from compiled app.js)
`LazySelect.fetchTasks` maps results to add `label = task_name + ' (' + project.name + ')'` before setting `this.options`. v-select renders using `label`. Any code setting `lazySelect.options` directly must call `labelledTask()` first, and the API request must include `with: ['project']`.

### `vSelectComp.select()` vs `lazySelect.inputHandler()`
- `inputHandler(id)` — updates parent form data only. v-select visual state does not update.
- `vSelectComp.select(option)` — triggers full chain: v-select → LazySelect → parent form. Both visual and data update. Use this for programmatic selection.

### `fetchTasks('')` returns nothing
The `onSearch` guard (`query.length >= 1`) exists because the backend returns no results for empty queries. Never call `fetchTasks('')` — use a direct `apiFetch` call with no search filter for initial task loading on focus.

### Placeholder detection via DOM
`lazySelect.value` is unreliable for detecting selected state. Use `!!lazySelect.$el.querySelector('.vs__selected')` — stable across Vue re-renders.

### MutationObserver re-entrancy rule
Any function called from a `MutationObserver` callback that writes to the DOM will trigger the observer again. Guard flags (`_fetching`, `currentStart`, etc.) must be set **before** the first DOM write, not after. Setting them after an `await` is too late — the re-entrant call arrives synchronously before the microtask queue drains.

### Cattr QueryHelper `where` format
`where: { field: value }` is parsed as `[$operator, $value]`.
- `field: ['=', [1, 2, 3]]` → `WHERE field IN (1, 2, 3)` (whereIn)
- `field: [1, 2]` → `WHERE field = 2` (misread: op=1, val=2) — WRONG
- `field: ['between', ['2026-01-01', '2026-01-31']]` → `WHERE field BETWEEN ... AND ...`

Always use the two-element `[$operator, $value]` format explicitly.

### Vue Router matched instances
`vm.$route.matched[n].instances.default` gives the component instance for a matched route level. More reliable than walking `$children`. Use this to read data properties from page-level components (e.g. `userIDs` from `TimeuseReport`).

**Search deepest-to-shallowest** (`matched.length - 1` down to `0`) — parent route components may have the same property name with a default empty value, and would be found first in a forward scan.

### AT-UI dropdown portal
AT-UI `<at-select>` dropdowns are appended directly to `<body>`, not inside the component's parent. CSS selectors scoped to a page element (e.g. `.time-use-report .at-select__dropdown`) will never match. Use `document.querySelectorAll('.at-select__dropdown')` globally.

### AT-UI multi-select selection commit timing
The AT-UI multi-select only commits the selection to the parent Vue component's reactive data when the **dropdown closes** (not on each option click). Reading `userIDs` from the Vue component while the dropdown is still open returns the previous selection. Always close the dropdown first (`document.body.click()`), wait at least one event-loop tick (100ms in practice), then read the value.

### Vue re-render on reload destroys injected containers
On page reload, Cattr's Vue app performs two render cycles: one early render before auth resolves, and a second after the user store is populated. The second render destroys and recreates the route component's DOM, removing any elements injected during the first render. Guard against this by resetting tracking state (e.g. `currentStart = null`) inside `injectContainer()` whenever the container is created — this forces `tick()` to trigger a fresh fetch rather than skipping because "nothing changed".
