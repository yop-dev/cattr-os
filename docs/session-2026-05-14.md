### BUG-019 — Web-owned sessions: double-logging / silent time loss

**Status:** ✅ Fixed and confirmed working.

**Discovery:** BUG-013 test (web start → web stop, ~6 minutes) produced two 3-minute intervals instead of one 6-minute interval. Root cause: both web bar AND desktop were trying to log intervals for the same session, conflicting with each other.

**Root cause — design conflict:**
- Desktop always pushes: gap interval (pre-detection) + periodic intervals (capture cycle)
- Web stop also tried to log: one big interval from `start_at` to `stop` (full session)
- These overlap entirely → double-counting if both succeed; silent time loss if web call is rate-limited (EAUTH502)

**Fix — desktop owns all interval logging:**
- `quick-create.js`: removed `owner === 'web'` interval creation from stop handler. Web stop now only calls `POST /api/tracking/stop` to clear the server cache.
- `web-sync.js`: changed `pushInterval = !_externalWebSession` → `pushInterval = true`. Desktop now always pushes the tail interval (last periodic → actual stop) on external stop, regardless of session owner.

#### Files Modified

| File | Repo | Change |
|---|---|---|
| `app/public/quick-create.js` | cattr-server | Removed interval creation from stop handler; web stop only clears session cache |
| `app/src/base/web-sync.js` | desktop-application | `pushInterval = !_externalWebSession` → `pushInterval = true` |

### BUG-020 — EAUTH502 Too Many Attempts (rate limiting)

**Status:** ✅ Fixed.

**Root cause:** Tracking routes were throttled at `throttle:120,1` (120 req/min). Web + desktop each poll `POST /api/tracking/current` every 1 second = 120 req/min total — at the exact limit. Any other API call (interval push, task sync) triggered 429 EAUTH502 errors.

**Fix:** Changed all three tracking routes (`tracking/current`, `tracking/start`, `tracking/stop`) in `api.php` from `throttle:120,1` to `throttle:600,1`. Rebuilt Docker image and redeployed.

### BUG-021 — Screenshots page shows only 1 of N screenshots for multi-interval session

**Status:** ✅ Fixed.

**Root cause:** `ItemController._index()` calls `$itemsQuery->paginate()` with no arguments → defaults to Laravel's 15-row page size. `perPage: 1000` in the request JSON body is silently ignored (`perPage` is in `QueryHelper::RESERVED_REQUEST_KEYWORDS`, skipped by WHERE processing, never passed to `paginate()`). On any date with 15+ total intervals across all users, the API returns only page 1 (15 rows), dropping everything after position 15.

**Fix:** Added `'X-Paginate': 'false'` to the `apiFetch()` headers. The `_index()` method checks for this header and switches from `paginate()` to `get()`, returning all matching rows without a page cap.

### BUG-023 — Desktop auto-start timer on task creation silently no-ops

**Status:** ✅ Fixed.

Creating a task+project from the desktop app showed the new task in the list but did not start the timer. Root cause: `tasks/create` IPC route sent a raw Sequelize model via `request.send(200, {task: createdTask})`. Electron's structured clone (used by `event.sender.send()`) only copies own enumerable properties — Sequelize stores the UUID `id` as a prototype getter, not an own property, so it's stripped. `result.body.task.id` was `undefined` in the renderer. `startTrack({ taskId: undefined })` silently fell into a no-op path.

**Fix:** Changed to `createdTask.dataValues` — a plain object with all fields as own properties.
