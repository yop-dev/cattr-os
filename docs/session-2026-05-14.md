### BUG-017 ‚Äî Hide all play buttons while tracking (task switching removed)

**Status:** ‚úÖ Fixed and confirmed working.

**Root cause:** `Task.vue` used `v-if="!active"` ‚Äî only hid the button on the currently active task row. All other rows kept their play buttons, allowing unreliable mid-session switches.

### BUG-019 ‚Äî Web-owned sessions: double-logging / silent time loss

**Status:** ‚úÖ Fixed and confirmed working.

**Discovery:** BUG-013 test (web start ‚Üí web stop, ~6 minutes) produced two 3-minute intervals instead of one 6-minute interval. Root cause: both web bar AND desktop were trying to log intervals for the same session, conflicting with each other.

**Root cause ‚Äî design conflict:**
- Desktop always pushes: gap interval (pre-detection) + periodic intervals (capture cycle)
- Web stop also tried to log: one big interval from `start_at` to `stop` (full session)
- These overlap entirely ‚Üí double-counting if both succeed; silent time loss if web call is rate-limited (EAUTH502)

**Fix ‚Äî desktop owns all interval logging:**
- `quick-create.js`: removed `owner === 'web'` interval creation from stop handler. Web stop now only calls `POST /api/tracking/stop` to clear the server cache.
- `web-sync.js`: changed `pushInterval = !_externalWebSession` ‚Üí `pushInterval = true`. Desktop now always pushes the tail interval (last periodic ‚Üí actual stop) on external stop, regardless of session owner.

#### Files Modified

| File | Repo | Change |
|---|---|---|
| `app/public/quick-create.js` | cattr-server | Removed interval creation from stop handler; web stop only clears session cache |
| `app/src/base/web-sync.js` | desktop-application | `pushInterval = !_externalWebSession` ‚Üí `pushInterval = true` |

### Timecard Export ‚Äî interval merging (display fix)

**Status:** ‚úÖ Done and confirmed working.

**Problem:** A 6-minute session logged as 3 desktop intervals (gap + periodic + tail) showed as 3 separate rows in Reports. User expected one row.

**Fix:** Added `mergeContiguousIntervals()` to `timecard-export.js`. Sorts intervals ascending, collapses any same-task + same-user intervals with a gap < 30 seconds into a single display row. Applied to both the HTML table and PDF export.

**Edit button for merged rows:** Stores `_firstEndAt`, `_lastId`, `_lastStartAt` on each merged row. `saveEdit()` makes two API calls for merged rows: edits first interval's `start_at` and last interval's `end_at`. All existing data retroactively merged at display time ‚Äî no DB changes needed.

### BUG-020 ‚Äî EAUTH502 Too Many Attempts (rate limiting)

**Status:** ‚úÖ Fixed.

**Root cause:** Tracking routes were throttled at `throttle:120,1` (120 req/min). Web + desktop each poll `POST /api/tracking/current` every 1 second = 120 req/min total ‚Äî at the exact limit. Any other API call (interval push, task sync) triggered 429 EAUTH502 errors.

**Fix:** Changed all three tracking routes (`tracking/current`, `tracking/start`, `tracking/stop`) in `api.php` from `throttle:120,1` to `throttle:600,1`. Rebuilt Docker image and redeployed.

### BUG-015 ‚Äî Screenshots project filter

**Status:** ‚úÖ Fixed.

**Investigation via DevTools:** Ran `Object.keys(inst).filter(k => k.includes('project'))` on the Screenshots component. Confirmed property name is `projectsList`, holding an array of project **objects** (not IDs). Also confirmed `where.project_id` in the API request was silently ignored ‚Äî `project_id` is not a column on the `time_intervals` table.

**Fix in `screenshots-grouped.js`:**
- `getSelectedProjectIds()` now reads `inst.projectsList` and maps `p => p.id`
- Removed `where.project_id` from API call; moved to client-side filter: `rows.filter(iv => iv.task?.project && projectIds.includes(iv.task.project.id))`

Confirmed working: select a project ‚Üí only that project's screenshots appear.

### UTC timestamps (BUG-016 design decision)

**Investigation:** `window.__cattrTz` was correctly set to `"America/Los_Angeles"` in the DB/PHP layer. However, the Dashboard's timer bar (green intervals) already displays in UTC. Making Screenshots show PDT would create a cross-page inconsistency.

**Decision:** UTC everywhere. Hardcoded `'UTC'` in:
- `screenshots-grouped.js`: `getCompanyTimezone()` returns `'UTC'`
- `dashboard-nav.js`: both `tz` variable assignments ‚Üí `'UTC'`

Screenshots timestamps now match the Dashboard timeline.

---

### Timezone selector removal

Since company timezone is always America/Los_Angeles and UTC display is now intentional, the timezone picker visible on Reports/Screenshots pages was redundant and potentially confusing.

**Investigation via DevTools:** The picker is `.controls-row .controls-row__item:last-child` across all pages ‚Äî consistent selector.

**Fix:** Added one CSS rule to `app/resources/views/app.blade.php`:
```css
.controls-row .controls-row__item:last-child { display: none !important; }
```

Timezone picker now hidden globally on all pages without touching Vue components.
