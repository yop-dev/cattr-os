### Bug fixed: timestamps displayed in UTC instead of company timezone

**Symptom:** Timecard table showed times like `06:42 PM` when the correct PDT time was `11:42 AM`.

**Root cause:** Cattr's API returns timestamps as `"2026-05-10 18:42:00"` — space-separated, no timezone marker. `new Date("2026-05-10 18:42:00")` in Chrome/V8 treats this as **local browser time** (PDT), not UTC. So `18:42` was interpreted as `18:42 PDT = 01:42 UTC`, then converting `01:42 UTC` back to PDT gave `06:42 PM` — wrong. The correct read: `18:42` from the API means `18:42 UTC = 11:42 AM PDT`.

**Fix:** `normTs()` normalizes API timestamps before `new Date()` is called:

```javascript
function normTs(s) {
    s = String(s || '').replace(' ', 'T');
    if (!/Z|[+-]\d{2}:\d{2}$/.test(s)) s += 'Z';
    return s;
}
```

Applied in `toLocalParts()`, `utcToLocalInput()`, and `durationSecs()`. This forces correct UTC parsing regardless of the API's timestamp format.

**Symptom:** Times sometimes showed in UTC on first page load.

**Root cause:** `getCompanyTimezone()` read from `vm.$store.getters['user/companyData'].timezone`. On initial render the Vue store hasn't populated yet — the getter returns `null`, fallback was `'UTC'`.

```php
@endphp

```

`getCompanyTimezone()` reads `window.__cattrTz` first (now always correct), with the Vue store as a fallback only.

### `localInputToUtcIso()` — round-trip UTC conversion

The `datetime-local` input yields a string like `"2026-05-10T18:42"` in the user's local timezone. Strategy: treat it as UTC, round-trip through `utcToLocalInput` to find the actual offset, then subtract it:

```javascript
function localInputToUtcIso(localStr, tz) {
    var asUtcMs    = new Date(localStr + ':00.000Z').getTime();
    var roundtrip  = utcToLocalInput(new Date(asUtcMs).toISOString(), tz);
    var roundtripMs = new Date(roundtrip + ':00.000Z').getTime();
    return new Date(2 * asUtcMs - roundtripMs).toISOString();
}
```

Works correctly for PDT (UTC-7) including DST boundaries. Fall-back ambiguity (one overlapping hour per year during fall-back) always picks the pre-transition occurrence — acceptable for a small internal team.

## Key Technical Notes

### Cattr API timestamp format
The API returns `"YYYY-MM-DD HH:MM:SS"` (space-separated, no timezone). Always call `normTs()` before `new Date()` on any API timestamp. This is not documented upstream — discovered through console logging the raw values.

### `CatEvent` post-save hooks are unreliable for custom edits
The `_edit()` helper in `IntervalController` fires the full filter/event pipeline after save. Any installed integration (GitLab, etc.) that subscribes to the after-action event will run — and if that integration is misconfigured it returns 500. Direct Eloquent `->save()` bypasses all of this safely.

### PHP timezone injection pattern
For any JS feature that needs the company timezone at render time, use `window.__cattrTz` injected via Blade. Reading from the Vue store at page load is unreliable — the store populates asynchronously after the initial render cycle. The DB query in Blade runs server-side before any JS executes.

### Cache-busting pattern for injected scripts
All custom scripts use `?v={{ filemtime(public_path('filename.js')) }}` in `app.blade.php`. This is the same approach Laravel Mix uses for compiled assets. No manual cache clearing needed after deploy.

## Part 2 — Timezone Display Fix (same day, later session)

### Problem discovered

After C-004 was deployed and tested, times on the Reports (Timecard Export) page showed **11:52 PM** for entries that the Team page and screenshot modal showed correctly as **6:52 AM**.

### Root cause: "two wrongs make right"

The Cattr desktop agent has a known bug: it sends the user's local time as if it were UTC. A 6:52 AM PDT session is stored as `06:52:00 UTC` in the DB instead of `13:52:00 UTC`.

Native Cattr pages (Team tab, screenshot modal) display timestamps **as stored** — they apply no UTC→local conversion. So `06:52 UTC` renders as `6:52 AM`, which looks correct even though the data is wrong.

Our `toLocalParts()` in `timecard-export.js` was doing the right thing — correctly converting `06:52 UTC` to `06:52 AM PDT` by applying the PDT offset. But since the stored value was already local time masquerading as UTC, the conversion produced the wrong display (`06:52 - 7h = 11:52 PM`).

**Fix:** Match native Cattr behavior — display timestamps as stored, without any UTC→local conversion. This will remain correct until IDEA-002 (desktop agent fork) fixes the agent to send real UTC.

### BUG-007 — Reports page user filter ignored on initial load

First tick saw `userIds = []`, fetched all-user data, set the guard keys. By the time the selection restored and triggered a second tick, the date guard saw no change and skipped the re-fetch.
