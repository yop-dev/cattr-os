**`EPERM` on re-package:** If `package-windows` fails with `EPERM: operation not permitted, unlink target\__uninstaller-nsis-cattr.exe`, delete that file manually and retry.

### C-020 — Clockify-Style Timer Bar (Web) + Web↔Desktop Bidirectional Sync

**Feature:** The quick-create dashboard bar was replaced with a full Clockify-style timer bar on the web, and the desktop app gained a 1-second polling module to stay in sync with the web.

**Backend — `app/routes/api.php`**

Added three inline-closure routes (no separate controller class — avoids Composer classmap regeneration in the upstream Docker image):
- `POST /api/tracking/start` — writes `{ task_id, owner: 'web', started_at }` to Laravel cache per user (TTL 24h)
- `POST /api/tracking/stop` — clears the cache entry
- `GET /api/tracking/current` — returns the current session or `{ active: false }`

**Frontend — `app/public/quick-create.js`** (full rewrite)

Idle state: task search with suggestions, project selector, Start button.  
Running state: task name (read-only) + project + live elapsed timer + Stop button.  
Polls `/api/tracking/current` every 1s — picks up start/stop from desktop within one cycle.  

Two UX bugs fixed after initial implementation:
1. `document` click listener fired immediately after focus opened the suggestion dropdown, closing it instantly — fixed with `stopPropagation` on the task input click event.
2. "No tasks yet" text stayed visible even after tasks loaded if the user clicked before the first fetch completed — fixed by re-rendering the dropdown inside `fetchTasks()` when input is already focused.

**Desktop — `app/src/base/web-sync.js`** (new file)

1-second polling module on the desktop side. Calls `GET /api/tracking/current`:
- If server reports a running session the desktop doesn't know about → calls `TaskTracker.start()` with the correct local task
- If server reports idle but desktop is tracking → calls `TaskTracker.stop()`
- If session owner is `'desktop'`, desktop→web direction is skipped (desktop is authoritative)

**Desktop — `app/src/routes/index.js`**

Fixed BUG-008 as part of C-020 integration: `webSync.startSync()` was only called on the `'authenticated'` event, which isn't emitted when the app starts with a saved token. Fixed by calling `startSync()` unconditionally at module load — it's idempotent (`if (pollTimer) return`).

### BUG-010 — 500 on Task Creation (fixed)

**Root cause:** `Task::boot()` observer in the upstream model calls `BelongsToMany::firstOrCreate` to add the task creator to the project's pivot table. When the user isn't already in the pivot, `firstOrCreate` finds no match and attempts to `INSERT` a new `User` row with only `['id' => ..., 'role_id' => ...]` — missing required columns (email, name, password, etc.) → SQL constraint violation → 500.

C-011 (all projects visible) made this common: employees could see and create tasks in projects they weren't pivot members of.

**Fix:** `app/app/Models/Task.php` — extracted from the running container and patched. Both `created` and `updated` observers changed from:
```php
$task->project->users()->firstOrCreate(['id' => $user->id], ['role_id' => ...]);
```
to:
```php
$task->project->users()->syncWithoutDetaching([$user->id => ['role_id' => ...]]);
```
`syncWithoutDetaching` only writes to the pivot table — it doesn't touch the `users` table at all.

## Key Technical Notes

### Electron 14 / Windows dev mode limitation
`require('electron')` in the main process resolves to `node_modules/electron/index.js` (returns binary path string) instead of the Electron API. This is specific to Electron 14 on Windows without the built-in intercept. Production builds (ASAR-packaged) work correctly. Dev mode requires Linux/macOS or Docker.

### yarn usage in this repo
Always use `node .yarn/releases/yarn-3.2.1.cjs <script>` — the yarn binary is bundled in `.yarn/releases/`. No global yarn installed on this machine.

### Build output
- `build/` — webpack renderer output (required before packaging)
- `target/` — electron-builder output (installer artifacts)

### `@cattr/node` project API
`api.projects` only exposes `list()` — no `create()`. Use `api.post('projects/create', { name })` directly. The `api.post()` method handles auth tokens, retries, and error wrapping. Response is `{ success, response: { data: { ...projectFields } } }`.

### IPC routes (renderer → main process)
- `tasks/create` — `{ name, projectId: [internalId], description }` — `projectId[0]` is SQLite UUID, not API externalId
- `tasks/sync` — syncs from server, returns `{ tasks, highlights }`
- `time/total` — returns total time for today
- `projects/sync` — syncs from server, returns `{ projects }`
- `projects/create` — `{ name }` — new route added this session, returns `{ project: dataValues }`

### Element UI locale
Default locale is zh-CN if no locale is passed to `Vue.use(Element, ...)`. Always pass `{ locale: ElementLocale }` where `ElementLocale = require('element-ui/lib/locale/lang/en')`.

### Vuex store — projects
`this.$store.getters.projects` returns all projects including external (GitLab etc.). Filter with `.filter(p => p.source === 'internal')` to show only Cattr-native projects. Use `String(project.id)` (SQLite UUID) as the select value and IPC `projectId[0]`.

### el-select vs el-autocomplete for project search + create
`el-select` with `filter-method`: reactive options re-renders (when computed properties update) cause el-select to reset its internal `query` state after each keystroke — unusable for typing new project names. `el-autocomplete` with `fetch-suggestions` uses a callback pattern: options are delivered once per query, no reactive DOM changes, input accumulates correctly. Use `el-autocomplete` whenever the list of options can change based on a store getter.

### Project UUID stability after sync
After `projects/sync`, `api.projects.list()` includes the newly created project, so `syncProjects` finds it by `externalId` and does an UPDATE (not DELETE+CREATE), preserving the SQLite UUID. Nonetheless, look up by `externalId` after sync as the safe pattern — it survives any future changes to sync behavior.
