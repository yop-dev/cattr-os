`app.js` used `Vue.use(Element)` with no locale option, defaulting to zh-CN. All Element UI default strings (e.g. "No matching data" in the `el-select` empty state) showed in Chinese. Fixed by importing `element-ui/lib/locale/lang/en` and passing it at init.

### `projects/create` IPC Route Added

`@cattr/node` has no `projects.create()` method (unlike `tasks.create()`). Added directly:
- **Controller** (`app/src/controller/projects.js`): calls `api.post('projects/create', { name, description: name, screenshots_state: 1 })` (API requires non-empty description and integer screenshots_state), creates SQLite record with `externalId`, `name`, `source`, `screenshotsState` from API response.
- **Route** (`app/src/routes/projects.js`): serves `projects/create`, returns `{ project: savedProject.dataValues }` on 200.
- **Renderer** (`TaskCreateModal.vue`): after project created, captures `externalId` from response, calls `projects/sync`, awaits `syncProjects` dispatch, then finds the project in the store by `externalId` to get the correct post-sync SQLite UUID.

### Build Pipeline — Windows

**The development pipeline (no dev server needed):**
```bash
cd C:\desktop-application

# 1. Compile renderer (Vue → build/)
node .yarn/releases/yarn-3.2.1.cjs build-production

# 2. Package Windows installer (→ target/)
node .yarn/releases/yarn-3.2.1.cjs package-windows
```

Output: `target/Cattr_Setup.exe` (NSIS installer) and `target/Cattr.exe` (portable).

**Why not `yarn` directly:** yarn is bundled in the repo at `.yarn/releases/yarn-3.2.1.cjs`. No global yarn installed. Use `node .yarn/releases/yarn-3.2.1.cjs <script>`.

**Why not dev mode:** Dev mode (`yarn dev-win`) does NOT work on Windows. The README explicitly says "Launch development version (Linux & macOS only)". Confirmed cause: Electron 14 does not intercept `require('electron')` in the main process when `node_modules/electron` is present locally — `require('electron')` returns the binary path string instead of the Electron API. This is a known limitation of Electron 14 on Windows without Docker.

**`postinstall` / `ffi-napi` failure is non-blocking:** `electron-rebuild` fails on `ffi-napi` (optional transient dep from `active-win`) because Visual Studio Build Tools are not installed. `ffi-napi` is optional (active window detection). `keytar` and `sqlite3` both have prebuilt binaries and work fine. The webpack build and electron-builder packaging both succeed despite this. Same error shows during `package-windows` native rebuild step — also non-blocking.

**`EPERM` on re-package:** If `package-windows` fails with `EPERM: operation not permitted, unlink target\__uninstaller-nsis-cattr.exe`, delete that file manually and retry.

## Key Technical Notes

### Electron 14 / Windows dev mode limitation
`require('electron')` in the main process resolves to `node_modules/electron/index.js` (returns binary path string) instead of the Electron API. This is specific to Electron 14 on Windows without the built-in intercept. Production builds (ASAR-packaged) work correctly. Dev mode requires Linux/macOS or Docker.

### yarn usage in this repo
Always use `node .yarn/releases/yarn-3.2.1.cjs <script>` — the yarn binary is bundled in `.yarn/releases/`. No global yarn installed on this machine.

### Build output
- `build/` — webpack renderer output (required before packaging)
- `target/` — electron-builder output (installer artifacts)

### `@cattr/node` project API
`api.projects` only exposes `list()` — no `create()`. Use `api.post('projects/create', { name })` directly. The `api.post()` method handles auth tokens, retries, and error wrapping. Response is `{ success, response: { data: { ...projectFields } } }`.

### IPC routes (renderer → main process)
- `tasks/create` — `{ name, projectId: [internalId], description }` — `projectId[0]` is SQLite UUID, not API externalId
- `tasks/sync` — syncs from server, returns `{ tasks, highlights }`
- `time/total` — returns total time for today
- `projects/sync` — syncs from server, returns `{ projects }`
- `projects/create` — `{ name }` — new route added this session, returns `{ project: dataValues }`

### Element UI locale
Default locale is zh-CN if no locale is passed to `Vue.use(Element, ...)`. Always pass `{ locale: ElementLocale }` where `ElementLocale = require('element-ui/lib/locale/lang/en')`.

### Vuex store — projects
`this.$store.getters.projects` returns all projects including external (GitLab etc.). Filter with `.filter(p => p.source === 'internal')` to show only Cattr-native projects. Use `String(project.id)` (SQLite UUID) as the select value and IPC `projectId[0]`.

### el-select filter-method vs allow-create
`allow-create` lets el-select manage its own input (no input reset) but cannot show a custom `+ Create "X"` label prefix. `filter-method` supports custom option labels but a `@visible-change` handler that resets the filter query will cause accumulated input to be lost. Solution: use `filter-method` with NO `@visible-change` handler.

### Project UUID stability after sync
After `projects/sync`, `api.projects.list()` includes the newly created project, so `syncProjects` finds it by `externalId` and does an UPDATE (not DELETE+CREATE), preserving the SQLite UUID. Nonetheless, look up by `externalId` after sync as the safe pattern — it survives any future changes to sync behavior.
